1.2 Các khái niệm cơ bản

📨 Message (Tin nhắn)

Đơn vị dữ liệu được truyền giữa các thành phần. Một message bao gồm:

  • Header: metadata (routing key, content-type, timestamp...)
  • Body: nội dung thực tế (JSON, XML, binary...)
  • Properties: TTL, priority, delivery mode...
📦 Queue (Hàng đợi)

Bộ nhớ đệm lưu trữ message theo thứ tự FIFO (First In, First Out). Queue đảm bảo:

  • Message không bị mất nếu consumer tạm thời không hoạt động
  • Kiểm soát tốc độ xử lý (backpressure)
  • Phân phối tải đều giữa nhiều consumer
📡 Topic (Chủ đề)

Kênh logic phân loại message theo chủ đề. Ví dụ:

  • orders.created – đơn hàng mới
  • orders.shipped – đơn hàng đã giao
  • payments.success – thanh toán thành công
  • inventory.low_stock – tồn kho thấp
🏭 Producer & Consumer

Producer (Publisher): Thành phần gửi message vào queue/topic.

Consumer (Subscriber): Thành phần nhận và xử lý message.

Hai bên không cần biết về nhau – đây là nguyên tắc loose coupling

Message Broker

Vai trò trung gian · Exchange · Routing · Phân phối message

2.1 Message Broker là gì?

Message Broker là thành phần trung gian nhận message từ Producer, lưu trữ và chuyển tiếp tới Consumer phù hợp. Broker giống như một bưu điện thông minh:

  • Nhận thư (message) từ người gửi (producer)
  • Phân loại, đóng gói, lưu trữ
  • Phân phối đúng địa chỉ (queue/consumer)
  • Xác nhận đã giao (acknowledgement)

2.2 Exchange – Trái tim của RabbitMQ

Trong RabbitMQ, Exchange là thành phần nhận message từ Producer và quyết định gửi tới Queue nào dựa trên routing key và binding.

Direct Exchange

Chuyển message tới queue có binding key khớp chính xác với routing key.

Ví dụ: routing_key = "error" → chỉ đến queue error_log

# Routing key "info" → queue_info
# Routing key "error" → queue_error
channel.exchange_declare(
  exchange='logs_direct',
  exchange_type='direct'
)
Fanout Exchange

Gửi message tới tất cả queue đã bind, bỏ qua routing key.

Ví dụ: Broadcast thông báo hệ thống tới mọi service

# Gửi tới TẤT CẢ queue đã bind
channel.exchange_declare(
  exchange='notifications',
  exchange_type='fanout'
)
Topic Exchange

Routing theo pattern với wildcard: * (1 từ) và # (nhiều từ).

orders.* khớp với orders.created, orders.shipped

# "*.critical" → queue_critical
# "orders.#" → queue_orders
channel.exchange_declare(
  exchange='app_events',
  exchange_type='topic'
)
Headers Exchange

Routing dựa trên attributes trong header message, không dùng routing key.

Ví dụ: format=pdf AND type=report → queue_pdf_reports

# Match theo header attributes
channel.exchange_declare(
  exchange='data_export',
  exchange_type='headers'
)

2.3 Ví dụ thực tế: Hệ thống Log

Một hệ thống logging dùng Direct Exchange để phân loại log theo mức độ:

## Ví dụ 2.1 – Direct Exchange (producer_log.py)
import pika, json

connection = pika.BlockingConnection(
    pika.ConnectionParameters('localhost')
)
channel = connection.channel()

# Khai báo Direct Exchange
channel.exchange_declare(
    exchange='logs_direct',
    exchange_type='direct',
    durable=True
)

def send_log(level: str, message: str):
    body = json.dumps({
        'level': level,
        'message': message,
        'service': 'order-service'
    })
    channel.basic_publish(
        exchange='logs_direct',
        routing_key=level,       # 'info', 'warning', 'error'
        body=body,
        properties=pika.BasicProperties(
            delivery_mode=2,    # persistent message
            content_type='application/json'
        )
    )
    print(f"Sent [{level.upper()}]: {message}")

# Gửi các loại log
send_log('info',    'User logged in: user_001')
send_log('warning', 'Inventory low: LAPTOP-X1 (5 left)')
send_log('error',   'Payment failed: order_12345')

connection.close()
📘 Lưu ý quan trọng
delivery_mode=2 làm cho message được lưu xuống đĩa (persistent). Nếu RabbitMQ crash và restart, message vẫn còn. Mặc định là delivery_mode=1 (transient – mất khi restart).

2.4 Dead Letter Queue (DLQ)

Khi message không thể xử lý được (lỗi, hết hạn, bị reject...), chúng được chuyển tới Dead Letter Queue:

## Ví dụ 2.2 – Cấu hình Dead Letter Queue
# Khai báo DLX (Dead Letter Exchange)
channel.exchange_declare(
    exchange='dlx', exchange_type='direct'
)
channel.queue_declare(queue='dead_letters', durable=True)
channel.queue_bind(queue='dead_letters', exchange='dlx')

# Queue chính với cấu hình DLX
channel.queue_declare(
    queue='orders',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'dlx',
        'x-message-ttl': 60000,  # 60 giây TTL
        'x-max-retries': 3
    }
)

Pub/Sub & Point-to-Point

Hai mô hình giao tiếp – So sánh – Use cases

3.1 Point-to-Point (P2P)

Trong mô hình Point-to-Point, mỗi message được xử lý bởi đúng một consumer. Queue đóng vai trò trung gian lưu trữ.

Producer
→
📦 Queue
→
Consumer 1 ✓
Consumer 2 (chờ)
Consumer 3 (chờ)

Mỗi message chỉ được nhận bởi một consumer (round-robin hoặc theo tải)

Use cases của Point-to-Point
  • Xử lý đơn hàng (mỗi đơn chỉ xử lý 1 lần)
  • Gửi email (mỗi email gửi 1 lần)
  • Task queue (resize ảnh, export PDF)
  • Thanh toán (tránh double charge)
  • Phân phối công việc cho worker pool
  • Hàng đợi in ấn

3.2 Publish/Subscribe (Pub/Sub)

Trong mô hình Pub/Sub, message được gửi tới một Topic, và tất cả subscriber đã đăng ký nhận một bản sao.

Publisher
→
📡 Topic
→
Email Service ✓
Inventory Service ✓
Analytics Service ✓

Mỗi subscriber nhận bản sao riêng của message

Use cases của Pub/Sub
  • Thông báo đặt hàng thành công (nhiều service cùng cần biết)
  • Cập nhật giá sản phẩm (nhiều frontend cần refresh)
  • Sự kiện hệ thống (audit log, metrics, alert)
  • Live chat / real-time notifications
  • Feed của mạng xã hội
  • Broadcast trạng thái game online

3.4 Ví dụ code: Pub/Sub với RabbitMQ Fanout

## Ví dụ 3.1 – Publisher (khi đơn hàng được tạo)
import pika, json

def publish_order_created(order_data: dict):
    connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
    channel = connection.channel()

    # Fanout exchange – gửi tới TẤT CẢ queue đã bind
    channel.exchange_declare(
        exchange='order_events',
        exchange_type='fanout',
        durable=True
    )

    message = json.dumps({
        'event': 'order.created',
        'data': order_data
    })

    channel.basic_publish(
        exchange='order_events',
        routing_key='',  # Fanout bỏ qua routing_key
        body=message
    )
    print(f"Published order event: {order_data['order_id']}")
    connection.close()

# Test
publish_order_created({'order_id': 'ORD-999', 'total': 500000})
## Ví dụ 3.2 – Subscriber 1: Email Service
import pika, json

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()

channel.exchange_declare(exchange='order_events', exchange_type='fanout', durable=True)

# Queue riêng cho Email Service (exclusive=True → tự xóa khi disconnect)
result = channel.queue_declare(queue='email_service_queue', durable=True)
channel.queue_bind(exchange='order_events', queue='email_service_queue')

def on_order_event(ch, method, properties, body):
    event = json.loads(body)
    order = event['data']
    print(f"📧 Email Service: Sending confirmation for order {order['order_id']}")
    # Gửi email thực tế ở đây...
    ch.basic_ack(delivery_tag=method.delivery_tag)  # Xác nhận đã xử lý

channel.basic_consume(queue='email_service_queue', on_message_callback=on_order_event)
print('Email Service listening...')
channel.start_consuming()

Bước 2: Pattern 1 – Hello World (P2P cơ bản)

## lab4_producer.py – Gửi message đơn giản
import pika

connection = pika.BlockingConnection(
    pika.ConnectionParameters(
        host='localhost',
        credentials=pika.PlainCredentials('admin', 'admin123')
    )
)
channel = connection.channel()

# Tạo queue (idempotent – gọi nhiều lần không bị lỗi)
channel.queue_declare(queue='hello', durable=True)

channel.basic_publish(
    exchange='',               # Default exchange
    routing_key='hello',       # Tên queue = routing key
    body='Xin chào RabbitMQ!',
    properties=pika.BasicProperties(delivery_mode=2)
)
print(" [x] Sent 'Xin chào RabbitMQ!'")
connection.close()
## lab4_consumer.py – Nhận và xử lý message
import pika, time

connection = pika.BlockingConnection(
    pika.ConnectionParameters(
        host='localhost',
        credentials=pika.PlainCredentials('admin', 'admin123')
    )
)
channel = connection.channel()
channel.queue_declare(queue='hello', durable=True)

# Prefetch count = 1: consumer chỉ nhận 1 message tại một thời điểm
channel.basic_qos(prefetch_count=1)

def callback(ch, method, properties, body):
    print(f" [x] Received: {body.decode()}")
    time.sleep(1)           # Giả lập xử lý
    ch.basic_ack(delivery_tag=method.delivery_tag)
    print(" [x] Done processing")

channel.basic_consume(queue='hello', on_message_callback=callback)
print(' [*] Waiting for messages. CTRL+C to exit')
channel.start_consuming()

Mở 2 terminal. Terminal 1 chạy consumer.py, Terminal 2 chạy producer.py. Quan sát message xuất hiện ở consumer. Thử chạy 2 consumer và 1 producer gửi 10 messages – nhận xét sự phân phối.

Bước 3: Pattern 2 – Work Queue (nhiều worker)

## lab4_work_producer.py – Gửi nhiều task
import pika, sys

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='task_queue', durable=True)

messages = [
    "Task 1: Resize image product_001.jpg",
    "Task 2: Generate PDF invoice_2025.pdf",
    "Task 3: Send email to 1000 users",
    "Task 4: Update search index",
    "Task 5: Compress video tutorial.mp4",
]

for msg in messages:
    channel.basic_publish(
        exchange='',
        routing_key='task_queue',
        body=msg,
        properties=pika.BasicProperties(delivery_mode=2)
    )
    print(f" [x] Sent: {msg}")

connection.close()
## lab4_worker.py – Worker xử lý task (chạy nhiều instance)
import pika, time, sys

worker_id = sys.argv[1] if len(sys.argv) > 1 else "W1"

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.queue_declare(queue='task_queue', durable=True)
channel.basic_qos(prefetch_count=1)  # Fair dispatch

def process_task(ch, method, properties, body):
    task = body.decode()
    print(f"[{worker_id}] Processing: {task}")
    time.sleep(len(task) / 20)    # Giả lập thời gian xử lý
    print(f"[{worker_id}] Done: {task[:30]}...")
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(queue='task_queue', on_message_callback=process_task)
print(f'[{worker_id}] Ready. CTRL+C to exit')
channel.start_consuming()

Bước 4: Pattern 3 – Topic Exchange (nâng cao)

## lab4_topic_producer.py – Gửi event theo topic
import pika, json, datetime

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.exchange_declare(exchange='events', exchange_type='topic', durable=True)

events = [
    ('orders.created.vn',   {'order_id': 'ORD-1', 'country': 'VN'}),
    ('orders.shipped.vn',   {'order_id': 'ORD-2', 'country': 'VN'}),
    ('orders.created.sg',   {'order_id': 'ORD-3', 'country': 'SG'}),
    ('payments.success.vn',  {'payment_id': 'PAY-1', 'amount': 500000}),
    ('payments.failed.vn',   {'payment_id': 'PAY-2', 'reason': 'insufficient funds'}),
]

for routing_key, data in events:
    channel.basic_publish(
        exchange='events',
        routing_key=routing_key,
        body=json.dumps(data)
    )
    print(f"Published [{routing_key}]: {data}")

connection.close()
## lab4_topic_consumer.py – Subscribe theo pattern
import pika, json, sys

# Nhận pattern từ argument: "orders.#", "*.created.*", "#"
binding_key = sys.argv[1] if len(sys.argv) > 1 else "#"

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()
channel.exchange_declare(exchange='events', exchange_type='topic', durable=True)

result = channel.queue_declare(queue='', exclusive=True)
queue_name = result.method.queue
channel.queue_bind(exchange='events', queue=queue_name, routing_key=binding_key)

print(f"[*] Subscribed to pattern: '{binding_key}'")

def on_event(ch, method, properties, body):
    data = json.loads(body)
    print(f"  → [{method.routing_key}] {data}")

channel.basic_consume(queue=queue_name, on_message_callback=on_event, auto_ack=True)
channel.start_consuming()

Bước 2: Các khái niệm Kafka

Topic

Kênh phân loại message. Tương tự folder trong filesystem. Ví dụ: user-events, order-created

Partition

Mỗi Topic chia thành nhiều Partition để scale ngang. Message trong 1 partition có thứ tự. Nhiều partition = nhiều consumer đọc song song.

Offset

Vị trí của message trong Partition. Consumer lưu offset để biết đọc tới đâu. Có thể seek về offset cũ để đọc lại.

Consumer Group

Nhiều consumer cùng group → chia nhau partition (scale out). Khác group → mỗi group nhận đủ tất cả message (Pub/Sub).

Broker

Một Kafka server. Cluster gồm nhiều broker. Mỗi partition có 1 leader broker và N replica brokers.

Retention

Kafka giữ message trong thời gian cấu hình (mặc định 7 ngày). Cho phép replay event khi cần debug hoặc migrate service mới.

Bước 4: Kafka Producer

## lab5_producer.py – Gửi events lên Kafka
from kafka import KafkaProducer
import json, time, random

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    key_serializer=lambda k: k.encode('utf-8') if k else None
)

def simulate_orders(count: int = 20):
    users = ['user_001', 'user_002', 'user_003']
    
    for i in range(count):
        order = {
            'order_id': f'ORD-{1000 + i}',
            'user_id': random.choice(users),
            'amount': random.randint(50000, 5000000),
            'timestamp': time.time()
        }
        
        # Key = user_id → đảm bảo cùng user vào cùng partition
        future = producer.send(
            topic='order-events',
            key=order['user_id'],
            value=order
        )
        
        record = future.get(timeout=10)
        print(f"Sent: {order['order_id']} → Partition {record.partition}, Offset {record.offset}")
        time.sleep(0.5)

simulate_orders(20)
producer.close()

Bước 5: Kafka Consumer

## lab5_consumer.py – Đọc events từ Kafka
from kafka import KafkaConsumer
import json, sys

group_id = sys.argv[1] if len(sys.argv) > 1 else 'analytics-group'
worker_id = sys.argv[2] if len(sys.argv) > 2 else 'C1'

consumer = KafkaConsumer(
    'order-events',
    bootstrap_servers=['localhost:9092'],
    group_id=group_id,
    auto_offset_reset='earliest',  # Đọc từ đầu nếu chưa có offset
    value_deserializer=lambda m: json.loads(m.decode('utf-8')),
    enable_auto_commit=True
)

print(f"[{worker_id}] Consumer group: {group_id} – listening...")

for message in consumer:
    order = message.value
    print(f"[{worker_id}] P{message.partition}@{message.offset}: "
          f"Order {order['order_id']} from {order['user_id']} – {order['amount']:,}đ")

6.1 Khi nào dùng gì?

Chọn RabbitMQ khi...
  • Cần routing linh hoạt (Direct/Topic/Headers Exchange)
  • Task queue với priority
  • Cần RPC (Request-Reply pattern)
  • Message phải được xử lý và xóa (không cần replay)
  • Team mới với Message Queue (UI dễ dùng)
  • Throughput vừa phải (< 100k msg/s)
Chọn Apache Kafka khi...
  • Cần throughput cực cao (> 100k msg/s)
  • Event sourcing – cần lưu lịch sử đầy đủ
  • Stream processing với Kafka Streams/Flink
  • Log aggregation từ nhiều service
  • Cần replay events (audit, debug, migration)
  • Hệ thống microservices phức tạp
🏢 Dùng trong thực tế
LinkedIn tạo ra Kafka để xử lý activity feed hàng tỷ events/ngày. Netflix dùng Kafka cho real-time monitoring và event sourcing. Grab/Gojek dùng cả hai: Kafka cho event streaming, RabbitMQ cho task processing.

  routing_key: 'payment.pending.{method}'
                              ↓
           ├─ 'payment.success.*' → Order fulfillment queue
           ├─ 'payment.failed.*'  → Retry + notification queue
           └─ 'payment.*.*'      → Audit log queue

6.3 Các Pattern nâng cao

Pattern 1: Saga Pattern (Distributed Transactions)

Thay vì 2-phase commit, dùng chuỗi events để đảm bảo tính nhất quán:

## Choreography-based Saga
# Bước 1: Order Service
publish('orders.created', order_data)

# Bước 2: Inventory Service (subscribe orders.created)
def on_order_created(order):
    if reserve_inventory(order):
        publish('inventory.reserved', order)
    else:
        publish('inventory.failed', order)  # Trigger rollback

# Bước 3: Payment Service (subscribe inventory.reserved)
def on_inventory_reserved(order):
    if charge_payment(order):
        publish('payment.success', order)
    else:
        publish('payment.failed', order)
        # Inventory Service cần subscribe payment.failed để hoàn lại

6.4 Best Practices tổng kết

✅ Nên làm
  • Dùng durable=True cho production queues
  • Luôn implement basic_ack / basic_nack
  • Cấu hình Dead Letter Queue cho error handling
  • Dùng idempotent consumers (xử lý duplicate message)
  • Monitor queue depth và consumer lag
  • Thiết kế message schema với versioning
  • Test với message được gửi nhiều lần (at-least-once delivery)
❌ Tránh làm
  • Đưa business logic vào message routing
  • Gửi message quá lớn (> 1MB)
  • Commit offset trước khi xử lý xong (Kafka)
  • Tạo quá nhiều queue nhỏ lẻ
  • Không có retry mechanism
  • Bỏ qua message ordering khi dữ liệu có dependency
  • Không monitor và alert queue depth
